# 锻炼完整训练系统设计

日期：2026-06-28

## 1. 背景

当前锻炼模块已经有动作列表、动作详情、完成组数、计划页、数据页和历史页，但后三个页主要是静态展示：

- 计划页能看到训练模板和计划卡片，但不能进入可执行训练。
- 数据页显示今日组数、时长、消耗和最高重量，但数据主要来自当前组数摘要或静态值。
- 历史页显示日历、趋势和历史条目，但不能打开真实训练详情。

用户明确选择“直接做完整训练系统”，目标是把锻炼模块从展示型页面升级为可执行、可记录、可复盘的闭环功能。

## 2. 目标

锻炼模块需要形成四段闭环：

1. 从计划开始训练。
2. 训练过程中记录完成情况。
3. 训练完成后生成历史记录。
4. 数据和历史页从真实训练记录中计算和展示。

第一版完整系统的成功标准：

- 计划页每个计划都能点击并打开详情。
- 计划详情能展示动作、总组数、预计时长、目标部位。
- 用户能从计划详情开始训练。
- 训练页进入计划训练模式后，只显示该计划动作。
- 完成训练后生成一条训练历史记录。
- 数据页能基于训练记录显示今日组数、时长、消耗、最高重量。
- 历史页能打开某次训练详情，并支持再次训练同计划。

## 3. 非目标

第一版不做以下内容：

- 不做云端同步训练计划。
- 不做多人协作或教练管理。
- 不做复杂健身算法推荐。
- 不做逐秒训练计时器。
- 不做动作视频。
- 不做完整运动科学评估。

这些能力以后可以加，但第一版先保证本地训练闭环稳定。

## 4. 用户体验设计

### 4.1 计划页

计划页从“静态卡片”升级为训练入口。

可点击对象：

- 训练模板：胸背日、核心日、恢复日、快练 10 分钟。
- 计划卡片：胸背强化、腿部稳定、核心恢复。

点击后打开计划详情弹层或详情页。

计划详情展示：

- 计划名称。
- 训练目标，例如胸背强化、腿部稳定、核心恢复。
- 动作数量。
- 总组数。
- 预计时长。
- 动作清单。
- 每个动作的次数、重量和备注。
- 操作按钮：开始训练、编辑计划。

第一版“编辑计划”先支持轻量编辑：

- 添加已有动作。
- 移除动作。
- 调整动作顺序。

暂不支持创建全新动作。

### 4.2 训练页

从计划开始训练后，锻炼模块进入“计划训练模式”。

页面行为：

- 顶部 tab 自动回到训练。
- 动作列表只显示当前计划内动作。
- 摘要卡展示当前计划进度，例如 `8/19 组`。
- 当前动作优先选中计划中第一个未完成动作。
- 用户仍然可以点动作进入详情并完成组数。

训练完成条件：

- 当前计划中的全部动作组数都达到目标组数。

完成后行为：

- 生成一条训练记录。
- 清空当前训练模式。
- 数据页立即反映本次训练。
- 历史页出现本次训练。

### 4.3 数据页

数据页从“展示固定数字”改为“读取训练记录统计”。

第一版统计项：

- 今日完成组数。
- 今日训练时长。
- 今日预估消耗。
- 今日最高重量。
- 最近 7 天训练次数。
- 最近 7 天完成组数。

点击数据卡后打开数据明细。

数据明细展示：

- 指标名称。
- 统计来源。
- 相关训练记录列表。
- 可跳转到历史详情。

### 4.4 历史页

历史页从“静态记录”改为真实训练历史。

历史页展示：

- 最近训练日历。
- 最近训练列表。
- 动作趋势卡。
- 进步趋势卡。

点击历史条目后打开训练详情。

训练详情展示：

- 训练日期和时间。
- 训练计划名称。
- 完成动作数。
- 完成组数。
- 预计消耗。
- 动作完成明细。
- 操作按钮：再次训练同计划。

### 4.5 与其他模块联动

第一版保留现有轻量联动，并让它来自训练记录：

- 健康模块读取今日锻炼组数。
- 计划模块生活事件流显示“完成锻炼”。
- 饮食模块可以从训练页继续进入，用于记录训练后饮食。

不在第一版里做复杂营养建议。

## 5. 数据模型设计

当前锻炼只有 `Map<String, int> workoutGroupsByAction`，它只能表达“某个动作完成了几组”，不足以支撑计划、历史和统计。

需要新增 3 个模型。

### 5.1 WorkoutPlan

训练计划。

字段：

- `id`：计划 ID。
- `name`：计划名称。
- `target`：目标说明。
- `bodyParts`：目标部位列表。
- `actionNames`：计划包含的动作名称列表。
- `estimatedMinutes`：预计时长。
- `createdAt`：创建时间。
- `updatedAt`：更新时间。

### 5.2 ActiveWorkoutSession

当前进行中的训练。

字段：

- `id`：训练会话 ID。
- `planId`：来源计划 ID。
- `planName`：来源计划名称。
- `startedAt`：开始时间。
- `actionProgress`：每个动作完成组数。
- `feedback`：训练反馈，例如轻松、适中、偏累。

`ActiveWorkoutSession` 只表示当前正在练的一次训练。

### 5.3 WorkoutHistoryEntry

已完成训练记录。

字段：

- `id`：历史记录 ID。
- `planId`：来源计划 ID。
- `planName`：来源计划名称。
- `startedAt`：开始时间。
- `finishedAt`：完成时间。
- `durationMinutes`：训练时长。
- `totalGroups`：完成总组数。
- `estimatedCalories`：预估消耗。
- `actionResults`：动作完成明细。
- `feedback`：训练反馈。

### 5.4 WorkoutActionResult

历史记录中的动作明细。

字段：

- `actionName`：动作名称。
- `bodyPart`：目标部位。
- `targetGroups`：目标组数。
- `finishedGroups`：完成组数。
- `reps`：次数。
- `weight`：重量。

## 6. 本地存储设计

第一版继续使用现有本地优先路线，数据存在客户端 SQLite。

新增表：

### 6.1 workout_plans

保存训练计划。

关键字段：

- `planId`
- `name`
- `target`
- `bodyPartsJson`
- `actionNamesJson`
- `estimatedMinutes`
- `createdAt`
- `updatedAt`

### 6.2 active_workout_session

保存当前进行中的训练。

只保存一条当前会话。

关键字段：

- `sessionId`
- `planId`
- `planName`
- `startedAt`
- `actionProgressJson`
- `feedback`

### 6.3 workout_history`

保存训练历史。

关键字段：

- `entryId`
- `planId`
- `planName`
- `startedAt`
- `finishedAt`
- `durationMinutes`
- `totalGroups`
- `estimatedCalories`
- `actionResultsJson`
- `feedback`

### 6.4 兼容现有 workout_groups

现有 `workout_groups` 继续保留，用于小组件和旧摘要兼容。

新的训练完成后，同时更新：

- `workout_history`
- `workout_groups`
- 应用级生活事件

## 7. 状态流设计

### 7.1 从计划开始训练

流程：

1. 用户点击计划。
2. 打开计划详情。
3. 用户点击开始训练。
4. 创建 `ActiveWorkoutSession`。
5. 切换到训练 tab。
6. 列表显示计划内动作。

### 7.2 完成动作组数

流程：

1. 用户进入动作详情。
2. 点击完成一组。
3. 更新 `ActiveWorkoutSession.actionProgress`。
4. 同步更新当前摘要。
5. 如果所有动作完成，提示完成训练。

### 7.3 完成训练

流程：

1. 判断当前计划动作全部完成。
2. 创建 `WorkoutHistoryEntry`。
3. 写入本地 SQLite。
4. 清空 `ActiveWorkoutSession`。
5. 推入生活事件。
6. 数据页和历史页刷新。

### 7.4 再次训练同计划

流程：

1. 用户在历史详情点击再次训练。
2. 根据历史记录的 `planId` 找到计划。
3. 创建新的 `ActiveWorkoutSession`。
4. 切换到训练 tab。

## 8. UI 组件边界

锻炼模块当前集中在 `workout_module.dart`。第一版允许继续在该模块内实现，但需要保持组件边界清楚。

建议拆分职责：

- 计划数据：计划模型和默认计划列表。
- 计划 UI：计划卡、计划详情、计划编辑。
- 训练 UI：计划训练状态、动作列表、动作详情。
- 数据 UI：统计卡、统计明细。
- 历史 UI：历史列表、历史详情。

如果文件继续变大，后续再把锻炼模块拆成：

```text
lib/modules/workout/
  workout_module.dart
  workout_models.dart
  workout_state.dart
  workout_sheets.dart
  widgets/
```

第一版实现时不强制拆文件，避免一次改动过大。

## 9. 错误和边界情况

需要处理以下情况：

- 计划里没有动作：详情页禁用开始训练。
- 动作资源图片丢失：沿用现有图片降级显示方式。
- 当前已有训练未完成：开始新计划前询问继续旧训练或放弃旧训练。
- 历史记录为空：历史页显示空状态。
- 数据记录为空：数据页显示 0 值和引导入口。
- 计划被编辑后，旧历史记录不跟随变化，保留当时训练明细。

## 10. 测试设计

第一版需要覆盖以下测试：

- 点击计划卡能打开计划详情。
- 计划详情点击开始训练后进入训练 tab。
- 计划训练模式只展示计划内动作。
- 完成全部计划组数后生成历史记录。
- 数据页根据训练记录展示今日统计。
- 历史页点击记录能打开详情。
- 历史详情点击再次训练能开启同计划训练。

这些测试优先写 widget test，因为当前锻炼模块主要逻辑仍在 Flutter 页面层。

## 11. 实现阶段

### 阶段 1：训练计划闭环

- 添加计划模型。
- 添加默认训练计划。
- 计划页可点击。
- 添加计划详情。
- 从计划详情开始训练。

### 阶段 2：训练会话

- 添加当前训练会话状态。
- 训练页支持计划训练模式。
- 完成组数写入会话。
- 判断计划完成。

### 阶段 3：历史记录

- 添加训练历史模型。
- 完成训练后生成历史。
- 历史页读取真实记录。
- 历史详情可打开。

### 阶段 4：数据统计

- 数据页读取训练历史。
- 数据卡可点击查看明细。
- 趋势卡从历史记录计算。

### 阶段 5：轻量编辑

- 计划详情支持添加已有动作。
- 支持移除动作。
- 支持调整动作顺序。

## 12. 验收标准

功能验收：

- 用户能从计划开始一次训练。
- 用户能完成训练并看到历史记录。
- 数据页不再依赖写死数字作为主要数据。
- 历史页不再只是静态条目。
- 健康和计划仍能收到锻炼组数联动。

质量验收：

- `flutter analyze` 无错误。
- 锻炼相关 widget test 通过。
- 原有模块切换测试通过。
- 不影响登录、计划、财务、饮食、健康入口。
